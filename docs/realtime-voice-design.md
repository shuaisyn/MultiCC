# 实时语音交互设计文档

## 目标

实现类似豆包的「像打电话一样」实时语音交互体验：
- **延迟目标**：首次响应 ≤2s（从用户说话结束到AI开始说话）
- **核心功能**：流式语音输出 + 用户打断机制
- **用户体验**：低延迟、自然流畅的对话体验

---

## 现状分析

### 当前语音能力

1. **语音输入 (STT)**
   - Web端：MediaRecorder 录音 → 上传 → Whisper API → 文本
   - APP端：AudioRecorder 录音 → 上传 → Whisper API → 文本
   - **问题**：录音完成才开始识别，延迟高（2-5秒录音 + 1-3秒STT）

2. **语音输出 (TTS)**
   - 当前只有 `speechSynthesis` 用于通知提醒
   - **缺失**：没有AI回复的语音输出能力

3. **流式语音识别（已有基础设施）**
   - `voice-asr.js`：支持 OpenAI Realtime / 火山引擎 / FunASR 的流式ASR
   - `public/voice-stream.js`：浏览器端 AudioWorklet 降采样 + WebSocket 流式传输
   - `public/voice-worklet.js`：实时 PCM16 降采样处理
   - **问题**：未被集成到主对话流程，只用于「语音输入」替代键盘

### 延迟瓶颈分析

| 环节 | 当前延迟 | 优化方向 |
|------|----------|----------|
| 用户说话 → 文本显示 | 2-5秒（录音完成后才上传） | **流式ASR**：说话时实时识别 |
| 文本发送 → AI开始响应 | 0.3-1秒 | 已是流式，优化不大 |
| AI响应 → 用户听到语音 | N/A（无TTS） | **流式TTS**：边生成边播放 |
| 用户打断AI说话 | N/A | **VAD打断**：检测用户说话时停止TTS |
| **总延迟** | **3-6秒** | **目标 ≤2秒** |

## 优化方案

### 方案概览

```
┌─────────────────────────────────────────────────────────────────┐
│                      实时语音对话流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [用户说话] ──AudioWorklet──→ [流式ASR] ──实时文本──→ [显示]    │
│       │                                       │                 │
│       │                                       ↓                 │
│       │                              [打断检测] ──→ 停止TTS    │
│       │                                       │                 │
│       └─────── VAD ─────→ [静音检测] ─→ [发送AI请求]           │
│                                                    │             │
│                                                    ↓             │
│  [用户听] ←── Audio播放 ←── [流式TTS] ←── [AI流式响应]        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心模块

#### 1. 流式ASR（已有，需集成）

**现状**：`VoiceStream` 类已实现，但未用于主对话流程。

**改进**：
- 集成到聊天输入框的「按住说话」模式
- 实时显示识别文本（partial → final）
- 添加 VAD（语音活动检测）自动发送

#### 2. 流式TTS（新增）

**方案选择**：

| 方案 | 延迟 | 成本 | 质量 | 推荐度 |
|------|------|------|------|--------|
| OpenAI TTS API | 高（整句生成后播放） | 高 | 优秀 | ⭐⭐ |
| OpenAI Realtime API | 极低（音频流） | 高 | 优秀 | ⭐⭐⭐⭐⭐ |
| Edge TTS (免费) | 中 | 免费 | 良好 | ⭐⭐⭐ |
| 火山引擎 TTS | 低 | 中 | 优秀 | ⭐⭐⭐⭐ |
| 浏览器 speechSynthesis | 低 | 免费 | 一般 | ⭐⭐ |

**推荐方案**：优先支持 OpenAI Realtime API（实时音频），其次支持 Edge TTS / 火山引擎作为备选。

#### 3. 用户打断机制

**实现**：
- 音频输入时持续运行 VAD（语音活动检测）
- 检测到用户说话 → 立即停止 TTS 播放 + 取消 AI 流
- 视觉反馈：打断时显示提示

#### 4. 静音检测自动发送

**实现**：
- 检测用户停止说话后 0.8-1.2 秒静音
- 自动发送已识别文本
- 避免用户手动点击发送

### 新增文件

1. **`public/voice-session.js`** - 语音会话状态机
   - 管理「听→思考→说→听」的循环
   - 协调 VoiceInput、VoiceOutput、VadMonitor

2. **`public/voice-output.js`** - 流式TTS播放器
   - 接收服务端TTS流
   - 音频队列管理
   - 支持中断

3. **`src/tts-service.js`** - 服务端TTS代理
   - 支持多种TTS后端
   - 流式返回音频

4. **`public/vad-monitor.js`** - VAD静音检测
   - 基于音量/能量的静音检测
   - 打断信号发射

## 性能目标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 首次响应延迟 | ≤2秒 | 用户说话结束 → AI开始说话 |
| ASR延迟 | ≤500ms | 用户说话 → 文字显示 |
| TTS首包延迟 | ≤300ms | 文本开始 → 音频开始播放 |
| 打断响应时间 | ≤100ms | 用户开口 → TTS停止 |

## 实现计划

### Phase 1: 流式ASR集成 (1-2天)

1. 将 `VoiceStream` 集成到聊天输入框
2. 添加「按住说话」按钮模式
3. 实现实时文本显示
4. 添加静音自动发送

### Phase 2: 流式TTS实现 (2-3天)

1. 实现 Edge TTS 服务端代理
2. 实现前端音频播放器
3. 连接AI回复 → TTS → 播放流程

### Phase 3: 打断机制 (1天)

1. 实现VAD静音检测
2. 用户说话时中断TTS
3. 视觉反馈

### Phase 4: 性能优化 (1-2天)

1. 延迟测试与优化
2. 音频缓冲策略调优
3. 降级方案（网络差时回退）
